java google code style

发表于 2020-08-10 更新于 2020-08-12

源文件规范

文件名

以.java结尾

编译方式

in UTF-8

特殊字符

1. 空格使用（0x20），ASCII horizontal space character，其他的whitespace被忽略（e.g. Tab)

2. 具有转移字符如（\b, \t, \n, \f, \r, \", \' 以及 \\），只有该表达方式，而不是用octal (e.g. \012) or Unicode (e.g. \u000a) escape.

源文件结构

源文件依次包含以下结构：
1. 许可证或者版权信息（optional）
2. 包声明
3. import声明
4. 有且仅有（exactly one）top-level class
5. 每一部分之间exactly one blank line（空行）

package statement

package statement不受column limit的影响

import statement

1、 import * 是不被允许的，一定得明确说明
2、 import顺序如下：

* 所有static import（静态导入）在一个block中
* non-static import在另一个block中
* 两个block中间用blank line分割

3、static import不能用在static nested class中，得用normal import的方式引入。

static nested class: a class created insie a class, it can not access non-static data and member.

类声明（class declaration）

1、 一个source file只能有一个top-level class
2、 一个类中相同方法必须放在一起
3、 类声明的顺序没有固定的recipe，但是必须logical

Formatting

Braces（花括号）

遇到选择时使用

braces在遇到if、else、for、do、while时候使用，即使body为空，或者只有一个statemnt

对于非空的block

遵循以下规则：

1、在 { 前不换行
2、在 { 后换行
3、在 } 前换行
4、在 } 后换行，当 } 后一个语句结束了。比如 跟在else或者冒号后面的 { 不换行。

对于空block

1、 可以同非空block一样对待，也可以直接用{}(concise)
2、但是在一个multi-block statment下，不允许用concise

block indentation(缩进)

每次缩进为2 spaces。

one statement per line

每个statement之后都有一个line break（断行）

列数限制：100

每一行最长有100个character（Unicode character point）。

Line-Wrapping

line wrapping：将一个长行分割成多行

where to break

1、如果在非赋值符号断开，那么断开处在符号之前，比如（. , ::, |)

2、如果在赋值符号处断开，那么断开处在符号之后，比如（=， foreach

3、一个函数名需要和之后的开括号“（”相邻

4、逗号（，）和之前的token相邻

5、箭头（->）之后断开

Indent

在line-wrapping中，后续的line必须比前一个line+4 indent
但是如果非第一行的相邻的两个line有着相似的语法结构，则用同样的缩进

whitespace（空格）

vertical whitespace

空行一般出现在

1、连续的类构造器或者成员之间，比如变量、构造函数、方法、嵌入类，、静态构造器以及实例构造器。

注意：连续的field之间可以没有空格。（但是如果field之间有逻辑关系，空格是必须的。

2、源文件格式中提到的空行要求。

3、同时可以根据提升可读性要求进行。比如在第一个members/initializer 的前面加空行，或者最后一个的后面。

horizontal whitespace

1、保留关键字和后面的括号之间需要空格。比如 if、for、catch和之后的“（”之间需要加空格。

2、保留关键字和之前的关闭花括号需要加空格。比如else、catch与之前的“}”需要加空格。

3、在所有的open花括号“{”之前加空格。几个例外：

* @SomeAnnotation({a, b})
* String[][] x = {{"foo"}};

4、在二元或者三元operator的左右两边，都要加空格。
以及operator-like symbols:

* & 
<T extends Foo & Bar>
* | 
catch (FooException | BarException e)
* ：
for statement之间的：
* ->
（String str) -> str.length()

除了（：：）比如 Object::toString不加空格
（.）比如 object.toString()

5、在, : ;的后面，或者”)”作为cast的后面

6、在注释符号的左右两侧 //

7、between a type annotation and [] or …

8、花括号“{ }”内侧左右两边的空格可加可不加。

horizontal alignment

在连续注释(//)中， 上下两行的注释对齐非必需

Grouping parentheses：recommended

head first design pattern

发表于 2020-01-31 更新于 2020-06-25

真正触动我想要系统地学习设计模式，是19年暑期在Google实习时，host指出我的代码没有考虑DI（dependency injection)。 由于之前一直都在学校内，项目上对代码规范要求低，只要实现功能即可。 所以当时现学现用的我，立下一个决心，一定要系统地过一遍设计模式。

尽管一个资深老大哥告诉我，最好的设计模式就是没有设计模式，但是对我这样一个即将步入职场的小菜鸟来说，掌握一些章法在心中，还是很有必要的！那么，我们就开始吧~~

欢迎来到设计模式的世界

设计模式的本质就是为了使得代码更加易于维护和扩展。

实际问题

有一个鸭子的抽象类


这时候有很多的鸭子，有些鸭子会飞会叫，有些鸭子会飞不会叫，有些鸭子会叫不会飞，如何实现quack()和fly()这两个方法？

解决方法

  1. 用继承
    把quack()和fly()放如抽象类。
    No：但是所有继承的子类都要去实现这些方法，包括不会叫和不会飞的。pass

  2. 定义quackable()和flyable()接口
    有该功能的对象去实现该接口。
    No：因为接口没有实现，所以每个子类都要单独实现，单独更改。代码无法复用

  3. 设计原则
    找出应用中可能需要变化之处，把它们独立出来，不要和那些不需要变化的代码混在一起。
    那么此时，就需要把flyable和quackable独立出来，声明成接口此时称为FlyBehavior和QuackBehavior。此时要记住另一个设计原则针对接口编程，而不是针对实现
    编程    。也就是说，之前对于接口的实现，是直接写在子类的，那么就相当于子类是针对实现编程，每一个实现由于在子类中，都被写得死死的，无法服用，也很难改动。所以一个很好的方法是就针对接口编程。另外写类分别实现FlyBehavior和QuackBehavior，称之为行为类。 由行为类而不是子类来实现行为接口。

这样的话，鸭子的子类就可以使用接口所表示的行为，实际的实现不会被绑死在鸭子的子类中。如下图：

实现鸭子的行为

不同的行为类去实现同一个接口。

这样的设计，可以让fly和quack的实现类被其他类复用，比如鸭鸣器，因为这些类已经与鸭子类无关了。
同时也可以新增一些行为，既不会影响到既有的行为类，也不会影响鸭子类。

这样的话，fly和quack行为就是一个接口，可以在运行时变化。那么行为实现类的赋值要在什么时候呢？如果在初始化变量的是偶，就类似面向实现编程。

解决方案：可以添加setFlyBehavior和setQuackBehavior类，动态改变行为。

观察者模式

有趣的事情发生时，可千万别错过了~ 这个模式就是帮咱们的对象知悉情况，不会错过感兴趣的事情。

这时候就需要用到观察者模式啦~ 让我们来看看到底什么是观察者模式~

认识观察者模式

就像是报纸订阅一样。你订阅了某家报纸，只要有新的报纸就会给你送过来，只要取消了，就停止送报。

而报社的业务就是出版报纸。

所以 观察者=出版者+报纸订阅
此时，subject=>出版者，observer=>订阅者。

流程： subject管理某些数据，一旦数据发生改变，新的数据会以某种形式送到观察者手上。某个对象可以选择订阅称为观察者，也可以随时取消订阅。

松耦合的威力

当两个对象之间松耦合，依然可以交互，但是并不清楚彼此之间的细节。

而观察者模式正式提供了一种对象设计，让subject与observer之间松耦合。

这也正式符合第三个设计原则：为了交互对象之间的松耦合设计而努力。

因为对于subject而言，唯一依赖的东西是一个对象列表，包含了所有实现observer接口的对象。所以有oberver的该变，只需要通过resgiterObserver()和removeObserver()方法改变对象列表即可。

所以可以独立地复用subject和observer。

以上解法，存在一个问题就是，observer的消息是被动接收，由subject push，但是如果有些obeserver不希望接受这么多消息呢？是否可以observer自己主动get？当然，此法也存在observer每次需要数据都得去pull的麻烦。

实现的思路：notify只是调用update函数，传的是weatherData整个引用，observer可以自己从weatherData中选取需要的数据。

可插拔

可以通过配置文件使得每个组件可“插拔”

总结

  • 观察者模式定义了对象之间一对多地关系。
  • subject用一个共同地接口来更新观察者。
  • subject和observer之间用松耦合地方式结合，彼此不知道互相的细节。

装饰者模式

blingbling地装饰对象吧~

讨论继承滥用的问题，以及如何使用对象组合的方式，做到在运行时装饰类。

继承泛滥的问题

比如starbuck在计划不同品种的饮品价格时，有一个超类beverage提供基础价位，然后每个子类继承它，同时单独计算其他配料的价格，如下图所示：


存在的问题是：

  • 如果有一个配料的价格需要修改
  • 如果需要新添或者减少一个配料
  • 如果需要新添一个饮品，但是某些配料根本用不上
    …

可见，继承很不灵活。继承设计子类的行为，是在编译时候决定的，所有的子类都会继承到相同的行为。然而！如果用组合的方式扩展对象的行为，就可以在运行时动态地进行扩展。

这时候，我们需要遵守地设计原则就是对扩展开放，对修改关闭

注意：我们无法让设计地每个部分都遵循开放-关闭原则。遵循开放-关闭原则通常会引入新地抽象层次，增加代码复杂度。我们需要把注意力集中在设计中最有可能改变的地方，然后运用开放-关闭原则。

认识装饰者模式

以上问题可以用装饰者模式来解决，用配料来递归装饰饮料。

定义：装饰者模式动态地将责任附加到对象上，若要扩展功能，装饰者提供了比继承更有弹性地替代方案。

Java I/O也是装饰者模式的一个体现

回到饮料的问题，主要的设计思路是，现有一个抽象类beverage，而后抽象装饰类继承beverage，意式浓缩和decaf继承beverage，奶泡、抹茶继承装饰类。之后就可以用奶泡抹茶去装饰意式浓缩和decaf。

注意：装饰者和被装饰者必须是一样的类型，也就是拥有共同的超类。目的是用继承达到类型匹配，而不是获得行为。这样才可以通过外嵌得到同样类型的饮料，毕竟espresso加上mocha之后还是饮料。

认识工厂模式

简单工厂模式

简单工厂其实不是一个设计模式，反而更像是一个编程习惯，但经常被使用。

所谓简单工厂模式就是有一个专门的factory类进行对象构建。需要构建类的时候，只要调用这个factory类即可。

但是让每个地区都有一个自己的factory对工厂的品质不好控制，所以采用工厂模式吧，把创建对象的方法写在子类里面，被调用。

工厂方法模式

工厂方法模式的核心就是在类中定义工厂方法，用来处理对象的创建，并将这样的行为封装在子类中。这样客户程序中关于超类的代码就和子类对象创建代码解耦了。

定义：工厂方法模式（Factory Method Pattern）定义了一个创建对象的接口，但由子类决定要实例化的类是哪一个。工厂方法让类把实例化推迟到子类。

所以，工厂方法模式和简单工厂模式的区别

  • 简单工厂模式有一个专门的factory类进行对象构建，工厂方法模式是定义一个create的抽象方法，让子类自己实现。
  • 工厂方法模式可以随时根据子类的需要改变对象构建的方法，但是简单工厂模式必须自己创建一个工厂类。不具备工厂方法的弹性。

DI依赖注入原则

然而！！如果一个pizzaStore直接根据type产生对应的pizza对象，那么，这是一个很依赖的Pizza店。
因为当你实例化一个对象的时候（也就是new！）就是在依赖它的具体类。一个Pizza店需要create不同的Pizza，所以有着很强的依赖性。

依赖倒置原则（Dependency Inversion Princile）：要依赖抽象类，不要依赖具体类

这个原则说明：不能让高层组件依赖低层组件，而且，不管高层或者低层组件，都应该依赖于抽象。

而用了工厂模式之后，PizzaStore只是依赖于Pizza这个抽象类，而不是具体的接口。由工厂方法（createPizza）来产生具体的类。

这时候，比萨店的设计变得很棒，具有弹性的框架，而且遵循设计原则。

但是如果有一个新的需求，不同地方同样的Pizza，所用的原材料虽然都是面粉、酱料、芝士等，但是是不同风味的，如果为不同的店创建不同风味的原材料呢？需要一个抽象原料工厂

抽象工厂模式

抽象工厂模式 提供一个接口，用于创建相关或者依赖对象的家族，而不需要明确指定具体类。

当然，你会发现在抽象工厂中，createDough()也是一种工厂模式的体现。

工厂模式是提供一个接口产生具体的类，抽象工厂模式是提供一个接口产生一个抽象类。而后由其他类实现该接口，被实际调用。

Pizza的代码利用相关工厂生产原材料，所生产的原料依赖所使用的工厂。Pizza不关心这些原料，只知道如何制作比萨。

总结：工厂模式是利用继承，通过继承父类的create方法，产生具体类。让类之间的依赖解耦。只存在在方法里面。
而抽象工厂模式就是利用组合，提供一个接口产生抽象类，由子类Factory进行具体的实现，可以将多个类组合。

总结

  • 所有工厂都是用来封装对象的创建。
  • 工厂方法使用继承，把对象的创建委托给子类，子类实现工厂方法来创建
  • 抽象方法使用对象组合，对象的创建被是实现在工厂接口所暴露出来的方法中。
  • 工厂方法允许类将实例化延迟到子类进行。

因为在多线程中，可能一个语句被多个线程执行，比如 i = i + 1, i初始为1，那么被连个线程同时执行期待结果为2，如果放在高速缓存中，（就是两个线程分别读取i的值放入各自所在地CPU的高速缓存中，然后线程进行1+1的操作，结果就有可能是1.
为了解决这个问题，有两种方式：

  1. 在总线上加LOCK地形式，来解决。因为CPU和其他不见及逆行通信都是通过总线来进行的，如果对总线加LOCK的话，也就是阻塞了其他CPU对其他部件的访问（比如内存）。效率低下。
  2. 缓存一致协议。（Intel MESI）保证每个缓存中使用的变量的副本是一致的。也就是说当CPU写数据的时候，如果发现操作的变量是共享变量，那么会发出信号通知其他的CPU将该变量的缓存置为无效状态。

并发编程中的三个概念

  1. 原子性：即一个操作或者多个操作，要么全部执行并且执行的过程不会被任何因素打断，要么就不执行。

java中只可以保证直接赋值是有效性的，其他不可以。
比如x = 10；而 y = x就无法保证原子性。

  1. 可见性：当做个线程访问一个变量时，一个线程改变了这个变量的值，其他线程能够立刻看到。

java中提供了volatile关键字来保证可见性。当一个共享变量被volatile修饰时，会保证修改的值立刻被更新到内存，而不是高速缓存。当有其他线程需要读取时，会去内存中读取。

而普通的共享变量不能保证可见性，因为普通共享变量被修改之后，什么时候被写入主存是不确定的，当其他线程去读取时，此时内存中可能还是原来的旧值，因此无法保证可见性。

  1. 有序性。
    即程序执行的顺序按照代码的先后顺序执行。

在Java内存模型中，允许编译器和处理器对指令进行重排序，但是重排序过程不会影响到单线程程序的执行，却会影响到多线程并发执行的正确性。

在Java里面，可以通过volatile关键字来保证一定的“有序性”（具体原理在下一节讲述）。另外可以通过synchronized和Lock来保证有序性，很显然，synchronized和Lock保证每个时刻是有一个线程执行同步代码，相当于是让线程顺序执行同步代码，自然就保证了有序性。

命令模式

封装调用：把方法调用（method invocation）封装起来。

命令模式的核心就是将动作的发起者和动作的执行者解耦。

场景

如果有一个遥控器，需要控制房间各个灯的开和关该任何设置？

封装所有房间的操作为execute接口，在调用的时候，传递对应的房间操作对象，执行对应的execute。

总结

  1. 命令模式：将请求封装成对象，可以让你使用不同的请求、队列，或者其他日志请求来参数化其他对象。命令模式也可以支持撤销操作。
  2. 命令模式将发出请求的对象和执行请求的对象解耦。
  3. 在被解耦的两者之间是通过命令对象（Command）进行沟通的，命令对象封装了接收者的一个或者一组动作。
  4. 宏命令是命令的一种简单延申，允许调用多个命令。

    宏命令：是一种批量处理的称谓。一般来说，宏是一种规则或者模式，或者语法替换。

最少知识原则

外观模式符合一个新的原则：最少知识原则：减少对象之间的交互，只留下几个密友。意味着当你在设计一个系统的时候，不管是任何对象，都要注意它所交互的类有哪些，并注意如何与这些类交互的，不要让太多类耦合在一起。

在一个对象的方法内，我们应该只调用以下范围的方法：

  • 该对象本身
  • 被当作方法的参数而传进来的对象
  • 该方法所创建或者实例化的任何对象
  • 对象的任何组件

问题描述

比如冲泡咖啡和茶的流程类似，都是将水煮沸、打开茶/咖啡包，倒入杯中，添加柠檬/牛奶和糖，那么在实现的时候，如果单独实现咖啡和茶两个类，就出现了代码重复。

模板方法定义了一个算法的步骤，并允许子类为一个或多个步骤提供实现。固定子类的算法流程，但是只专注在算法本身，而由子类提供完整的实现。可以将代码的复用最大化。

总之，模板就是要给方法，将算法定义成一组步骤，其中的任何步骤都可以是抽象的，由子类负责实现。

hook

钩子：是一种被声明在抽象类中的方法，但只有空的或者默认的实现。钩子的存在，可以让子类由能力对算法的不同点进行挂钩。

Q&A

问： 创建一个模板方法地时候，如何决定使用抽象类或者钩子？
答： 如果子类必须提供算法中地某个方法或者步骤的实现，就用抽象方法。如果算法的这个部分是可选的，就用钩子。

问： 使用钩子的真正目的是？
答： 可以让子类实现算法中可选的部分。

总结

  • 模板方法模式：在一个方法中定义一个算法的股价，而将一些步骤延迟到子类。使得子类可以在不改变算法结构i的情况下，重新定义算法中的某些步骤。
  • 模板方法的抽象类可以定义具体方法、抽象方法和钩子
  • 抽象方法由子类实现
  • 钩子是一种方法， 它在抽象类中不做事，或者只做默认的事情，子类可以选择要不要去覆盖它。

状态模式

记录对象当前的状态，从而控制对象的行为（类似自动机）

问题引入

一个自动售货机，有 “无硬币，有硬币，售出饮料，售罄” 等状态，如何去实现这样一个自动售货机？
![state]state.png

如果只是简单记录状态，根据不同的变化条件跳转状态的话（以条件为准定义insertQuater(),ejectQuater(),turnCrank()函数），一旦有新增的状态，代码需要变更地方大，难以维护。

所以应该为每个状态新建一个类，每个类中实现状态对应的条件跳转即可。

以上做法的好处是：

  • 将每个状态的行为局部化自己的类中
  • 将容易产生问题的if语句删除
  • 让每个状态“对修改关闭”，对”扩展开放”

状态模式将状态封装成独立的类，并将动作委托到代表当前状态的对象。允许对象在内部状态改变时，看起来好像修改了它的类。

定义代理模式

代理模式为另一个对象提供要给替身或者占位符以控制对这个对象的访问。远程代理是一般代理的一种实现。

  • 远程代理控制访问远程对象
  • 虚拟代理控制访问创建开销大的资源
  • 保护代理基于权限控制对资源的访问。

复合模式

复合模式（Compound Pattern）在一个解决方案中结合两个或者多个模式，以解决一般或者重复发生的问题。

与设计模式相处

设计原则

  • 找出应用中可能需要变化之处，把它们独立出来，不要和那些不需要变化的代码混在一起。
  • 针对接口编程，而不是针对实现编程。
  • 为了交互对象之间的松耦合设计而努力。
  • 多用组合，少用继承。
  • 类应该对扩展开放，对修改关闭。
  • 最少只是原则：减少对象之间的交互，只留下几个密友。
  • 好莱坞原则：将决策权放在高层，以便决定和以及何时调用低层模块（模板方法模式）
  • 单一责任原则：一个类应该只有一个引起变化的原因（也就是尽量让每个类保持单一责任，高内聚）
